# S0.10 — Human-run setup (the M0 finish line)

> Everything in M0's automated slices is done. The remaining M0 work is infrastructure and
> account setup that only you can do (it needs credentials, DNS, a cloud console, and repo-admin
> rights). This checklist is assembled from what each slice reported. Nothing here is blocked on
> more code — but the M0 demo (`docker compose up` in prod, CI green end-to-end, a real deploy)
> can't be exercised until it's done.
>
> Order matters only where noted. Most of these are independent.

## 1. AWS — before the first `terraform apply`

The Terraform (`infra/terraform/`) provisions the SQS grading queues and a GitHub OIDC role. It is
written but **nothing is applied** — no AWS resource exists yet.

- [ ] **Check whether the GitHub OIDC provider already exists** in the AWS account:
  ```sh
  aws iam list-open-id-connect-providers \
    --query "OpenIDConnectProviderList[?contains(Arn, 'token.actions.githubusercontent.com')].Arn" --output text
  ```
  - Prints nothing → leave the Terraform defaults (it will create the provider).
  - Prints an ARN → set in `infra/terraform/terraform.tfvars`: `create_github_oidc_provider = false`
    and `existing_github_oidc_provider_arn = "<that ARN>"`. **If you skip this and the provider
    already exists, `terraform apply` fails half-way — after the four queues are created — leaving a
    partial stack.** AWS allows exactly one provider per URL.
- [ ] **Create the S3 state bucket by hand** (Terraform can't create the bucket that holds its own
  state). The four `aws s3api` commands (create in eu-west-3, enable versioning, block public
  access, enable encryption) are in `infra/terraform/README.md`. Then set the bucket name in
  `infra/terraform/backend.tf` (replacing the `REPLACE_ME-keepup-tfstate` placeholder — a bucket
  *name* is not a secret; its contents are).
- [ ] `cd infra/terraform && terraform init && terraform plan && terraform apply`.
- [ ] Note the outputs: the **role ARN** and **region** (both non-secret). There is deliberately **no
  long-lived AWS access key** anywhere — CI authenticates via OIDC.

## 2. GitHub repo settings

- [ ] **Repository variables** (Settings → Secrets and variables → Actions → *Variables* — these are
  not secrets): `CI_ROLE_ARN` (the role ARN from step 1) and `AWS_REGION` (`eu-west-3`). Until
  `CI_ROLE_ARN` is set, the `sqs-contract` job skips itself (fails closed).
- [ ] **Create a GitHub Environment named exactly `ci`** (Settings → Environments). The AWS role's
  trust policy scopes its subject to `repo:RoTour/keepup:environment:ci`, so the `sqs-contract`
  job's `environment: ci` is what makes the minted OIDC token match. Case-sensitive — it must be
  `ci`. (Optional: add environment protection rules later.)
- [ ] **Deploy secret + variable** (for the `deploy` job, once the Coolify webhook exists — step 3):
  repo secret `COOLIFY_DEPLOY_TOKEN` and repo variable `COOLIFY_DEPLOY_URL`. Until both are set,
  `deploy` is a no-op. (The workflow sends the token as an `Authorization: Bearer` header, not in the
  URL.)
- [ ] **Branch protection.** This is where the *path-filter trap* lives — read `docs/WORKFLOW.md`
  §3.2 before configuring. Short version:
  - `master`: no direct pushes; fast-forward-only from `staging`; require checks green.
  - `staging`: no direct pushes; PR required; squash-merge (linear history).
  - Required checks: `secrets-scan` is always safe to require (it isn't path-filtered).
    `backend-test`, `frontend-test`, `raw-html-gate` **are** path-filtered — do **not** mark them
    required as-is, or a docs-only PR hangs forever waiting on a check that never runs. Either adopt
    the aggregate `all-checks-passed` gate (preferred — see §3.2) or drop the path filters first.
- [ ] **GHCR package visibility.** On the first `build-push` run, `ghcr.io/rotour/keepup-backend`
  and `-frontend` are created **private**. This is a build-in-public project — decide whether to make
  the packages public and link them to the repo (Settings → Packages).

## 3. Coolify / VPS

- [ ] **Check memory headroom first:** `ssh hostinger2026 free -m`. The box (16 GB) runs two Supabase
  stacks + judge0 + others; keepup adds staging + prod (~1.2–1.5 GB each) + RabbitMQ (~250 MB). 16 GB
  should be ample, but confirm before committing.
- [ ] **RabbitMQ container** (Coolify), one instance, **two vhosts**: `staging` and `prod`. Both
  environments run `KEEPUP_QUEUE=rabbit` against it. Create an app user per vhost (not `guest`).
- [ ] **Two Postgres databases** on the existing Supabase Postgres: `keepup_staging` and
  `keepup_prod`, each with **its own owning role** scoped to just that database. **Connect directly
  to the Postgres container (port 5432, session mode) — never through Supavisor (6543).** ADR-0021
  and `docs/WORKFLOW.md` §1.1 explain why: a transaction-mode pooler silently kills `LISTEN/NOTIFY`
  and session advisory locks, and the failure mode is *nothing happening*, mid-class. Use
  `scripts/db-tunnel.sh` for local access.
- [ ] **Two Coolify apps** (staging, prod) deploying `ghcr.io/rotour/keepup-backend:<sha>` — the
  **same image digest** is promoted from staging to prod, never rebuilt. Plus the frontend image.
  Per-role instances: `web ×2, worker ×1, relay ×1` via `KEEPUP_ROLES` (see below). Caddy
  reverse-proxies `/` → frontend, `/api` + `/ws` → backend web **with WebSocket upgrade**.
- [ ] **Environment variables per app** (all runtime, never baked into an image):
  `DATABASE_URL` (direct 5432), `KEEPUP_ROLES` (per instance), `KEEPUP_QUEUE=rabbit`, the RabbitMQ
  URL/creds, `OPENROUTER_API_KEY` (separate key per environment, with a spend cap), SMTP creds
  (prod). The `migrate` one-shot runs before the app instances.
- [ ] **Coolify deploy webhook** per app → put the URL/token into GitHub as `COOLIFY_DEPLOY_URL` /
  `COOLIFY_DEPLOY_TOKEN` (step 2). Webhook mechanics were an open item — confirm how Coolify pins a
  specific image digest on trigger.

## 4. The M0 demo (what "done" looks like)

Once the above is in place, M0's demo criterion is exercisable for real:
- `docker compose up` locally boots the app in all three roles (already true locally).
- CI is green end-to-end on a PR, and a push to `staging` builds + pushes the image and deploys it.
- ArchUnit is enforcing (already true).

After that, M1 begins with the trainer-identity contract slice.
